# 初版开发计划

## 1. 项目目标

先做出一个可维护、可逐步上线的第一版平台，覆盖以下主链路：

- 面向客户的 API 交付页面
- 面向内部的资源调度与部署页面
- 下游机器上的 SGLang 推理服务编排
- API 使用量统计与计费

这个阶段不是追求“大而全”，而是先把业务闭环跑通。

## 2. 阶段计划

### Phase 0：项目基础

目标：

- 建立仓库结构
- 固定技术方案
- 补齐核心设计文档

交付物：

- monorepo 目录布局
- 架构总览文档
- MVP 路线图
- 初版执行计划

说明：

- 这一阶段主要解决“以后怎么不乱”的问题
- 先把边界定清楚，再开始写代码

### Phase 1：工程脚手架

目标：

- 让仓库具备本地开发和持续扩展能力

交付物：

- workspace 管理
- TypeScript 基础配置
- lint / format 配置
- portal、admin、api-server 三个应用骨架
- 环境变量规范

说明：

- 这一阶段不要求业务完整
- 重点是把工程运行底座搭起来

### Phase 2：账号与访问能力

目标：

- 先打通客户和管理员的登录/鉴权链路

交付物：

- users / tenants 基础 schema
- auth 模块
- 角色权限检查
- API Key 管理

说明：

- 后面几乎所有能力都依赖身份、租户和权限
- API Key 也是运行时网关的前置条件

### Phase 3：模型与部署管理

目标：

- 让管理员能管理模型、机器并发起部署

交付物：

- 模型注册表
- 节点注册
- 部署任务 API
- scheduler worker
- SGLang 部署集成

说明：

- 这里是你的平台核心竞争力之一
- 不只是“创建任务”，还要考虑重试、幂等、资源释放

### Phase 4：运行时请求链路

目标：

- 让客户可以真正调用到一个可用的推理服务

交付物：

- runtime gateway
- 运行时鉴权与限流
- 模型路由
- 请求日志

说明：

- 到这一阶段，平台开始具备真实对外能力
- 调用链路能跑通比 UI 完善更重要

### Phase 5：计量与计费

目标：

- 完成业务闭环，做到“调用可计量、费用可结算”

交付物：

- usage event 入库
- 成本计算
- ledger 账本记录
- 客户账单页面

说明：

- 原始 usage 数据必须先落库
- 账单计算应基于 usage 派生，而不是反过来

### Phase 6：稳定性加固

目标：

- 提高可运维性和可靠性

交付物：

- 健康检查
- 部署重试策略
- 告警
- 审计日志加强
- 调度与计费模块基础测试

说明：

- 这一阶段是从“能跑”走向“能稳定跑”

## 3. 当前最优先任务

下面这些是下一步最值得做的具体事项：

1. 初始化 workspace 工具链
2. 搭建 `apps/web-portal`
3. 搭建 `apps/web-admin`
4. 搭建 `apps/api-server`
5. 增加 Prisma schema 和首批 migration
6. 增加 auth、tenant、API key 模块

## 4. 设计规则

- 即使先放在一个后端服务里，也要保持领域模块分离
- 运行时 API 和控制台 API 要分开设计
- 部署任务和计费流程优先按幂等方式实现
- 先记录原始 usage event，再做账单聚合
- 不要过早拆成微服务

这些规则的目的只有一个：避免后面返工。

## 5. 初始里程碑的完成标准

满足以下条件时，可以认为第一个开发里程碑完成：

1. 仓库可以本地安装并运行
2. 后端可以连接数据库
3. portal 和 admin 可以渲染基础页面壳子
4. auth 和 API key 模块端到端打通

## 6. 额外提醒

- 你的平台本质上不是简单网站，而是“小型 AI 资源与计费平台”
- 最容易低估的是调度复杂度和计费一致性
- 最容易写乱的地方是把页面逻辑、领域逻辑、数据库逻辑混在一起

因此，当前阶段最重要的不是页面多漂亮，而是：

- 目录要清楚
- 模块要分层
- 核心链路要能闭环
